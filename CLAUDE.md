# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This repository is for authoring and collecting agent skills focused on research workflows, industrial R&D, paper writing, PhD applications, language test preparation, presentation/deck building, and implementation-heavy coding work. Skills should help agents support research ideation, literature grounding, industry problem framing, solution design, pilot evaluation, implementation, reproducibility, paper writing, reviewer response, artifact release, research-oriented applications, IELTS preparation, slide/deck production, and coding support.

Coding-support skills are allowed only when they encode something a capable agent does *not* already do by default; generic craft guidance (style, testing patterns, standard debugging) is explicitly out of scope and was removed from this repo. Generic README, architecture, slide, or meta-discovery skills should not be added unless they directly support research, industrial R&D, paper production, PhD applications, language test preparation, presentation/deck building, or coding-heavy research work.

## Structure

- `skills/` contains individual skills. Each skill should live in its own directory and include a `SKILL.md` file.
- `DESIGN-PHILOSOPHY.md` records the repo's durable skill-design rationale and supporting references. When adding, removing, or materially changing skill-family philosophy, learning/audit protocols, role boundaries, or evidence-backed design claims, update this file if the change affects the rationale.
- Skills are filed at `skills/<category>/<skill>/SKILL.md` ÔÇö **at most two levels under `skills/`**. This is not cosmetic: the skills CLI's default discovery stops at that depth, so a skill nested deeper is invisible to anyone installing without `--full-depth`. Do not create sub-categories inside a category; group siblings with a name prefix (`theory-*`, `apply-*`, `banking-*`, `flow-*`) instead.
- `skills/init-workspace/` sits at the top level *deliberately*, as the entrypoint skill for starting any new project. Top-level placement is reserved for that role; everything else belongs in a category.
- `skills/research-discovery/` contains early-stage research framing skills, plus `gap-finder` (mining others' papers for one's own research gaps ÔÇö discovery work, not claim-checking; it also owns the weekly lemma-autopsy habit: weaken one hypothesis, predict the first failed proof line, construct the smallest witness, and only then compare with the authors' discussion). It also holds `cross-domain-analogy-finder`, a high-variance foraging skill that finds a concept's structural twin in a distant field ÔÇö either to borrow a portable solution (offense) or to check whether an idea already exists under another name (defense) ÔÇö gated by a four-way classifier (identical / portable / twin-with-a-delta / surface-only) and an import-a-result test; it grounds prior-art claims through `literature-triangulation` and hands a twin's residual mismatch to `gap-finder`. It is for distant-field structural transfer, not near-field paper search or area survey (both `literature-triangulation`) or stress-testing one chosen idea (`research-idea-stress-test`). The directory also holds `gap-motivation-builder`, the Socratic bridge between finding a gap and proposing a solution: it takes one gap (from `gap-finder` or anywhere else) and interrogates the five-link motivation chain (consequence Ôćĺ beneficiary Ôćĺ blocker Ôćĺ timeliness Ôćĺ payoff), refuses "understudied" as motivation (gap-spotting vs problematization), runs the why-hasn't-this-been-done trichotomy (trivial / done under another name / genuine blocker), and converges on a verdict (MOTIVATED / CONDITIONALLY MOTIVATED / WEAK / UNMOTIVATED / NOT A GAP) plus solution desiderata ÔÇö requirements any future method must satisfy, each traced to a chain link, including at least one anti-desideratum ÔÇö while never proposing the method itself. Keep the pipeline boundaries: `gap-finder` finds gaps, `gap-motivation-builder` motivates one and derives desiderata, `research-idea-stress-test` attacks the chosen solution direction, `paper-idea-and-scope-brainstormer` scopes the paper, and `abstract-and-intro-writer` turns a finished motivation into prose.
- `skills/research-evidence/` contains literature mapping, Related Work writing, standalone survey/tutorial/perspective paper writing, and citation verification skills. Keep the boundary clear: `related-work-writer` is for Related Work sections inside original research papers, while `survey-paper-writer` is for papers whose main contribution is the synthesis/taxonomy/tutorial/roadmap itself.
- `skills/research-experimentation/` contains experiment design, benchmark/baseline selection, statistical testing, and evaluation skills. `experiment-design` covers both halves of the empirical loop: designing the minimal decision-relevant experiment *and* diagnosing a run that already went wrong (symptom classification Ôćĺ data-path audit Ôćĺ mechanism-wise tuning), plus scaling-curve fitting via its bundled `scripts/fit_scaling_law.py`. The former `deep-learning-experimentation/` sub-family (data-pipeline-auditor, training-run-forensics, hyperparameter-triage, experiment-trace-summarizer, representation-probe-designer, scaling-law-tracker) was folded into it ÔÇö do not re-add those as separate skills; they were ~200-word restatements of default behavior.
- `skills/research-engineering/` contains implementation recipes for research experiments (PyTorch, JAX) plus `research-codebase` for structuring a whole single-paper codebase. It also holds `paper-to-code`, which reproduces a paper's method in minimal single-file PyTorch when no official code exists, to check whether the method works as claimed ÔÇö one falsifiable claim, an assumptions ledger for underspecified details (backed by `references/underspecified-defaults.md`), staged verification gates (mechanics Ôćĺ overfit tiny batch Ôćĺ directional method-vs-baseline test), and an explicit works/doesn't/inconclusive verdict. It is for sanity reproduction, not for designing your own training setup (`pytorch-training-recipe`/`jax-training-recipe`) or auditing a released artifact (`reproducibility-audit`).
- `skills/research-reasoning/` contains theorem, claim, proof, and theory-heavy skills (`gap-finder` now lives in `research-discovery`). Keep the two claim-checkers distinct: `argument-audit` is the domain-general, interactive/Socratic "does the stated evidence actually establish this claim" checker (math, essay, or everyday argument) that walks the user to the weak link one question at a time and converges to a validity verdict + smallest fix ÔÇö whereas `theorem-and-claim-audit` is the direct, report-style audit specialized to formal math/ML claims (verdict handed over, not Socratic), and it now also owns step-by-step derivation checking (the former theory-derivation-auditor, merged in: name the rule justifying each step, flag silent assumptions, mark the *first* invalid step). Route general "is this reasoning valid" questions to `argument-audit`, heavily mathematical/empirical paper claims wanting a written audit to `theorem-and-claim-audit`, and external "does the cited source actually say this" to `citation-auditor`.
- `skills/learning-and-metacognition/` contains the teaching and metacognitive role family. `professor-mentor-technical-teaching` teaches fresh material; `concept-exercise-generator` builds graded verification practice; `knowledge-debt-audit` audits ownership of AI-assisted load-bearing steps and schedules variable-cue re-probes after repayment; `naive-student` lets the user repay or verify understanding by teaching an honest novice whose confusion must be earned from the actual explanation; `whiteboard-peer` co-solves the unfinished middle with committed partial attempts and real disagreement but never takes over the full solution; and `professor-critic` grades a finished artifact against a named reader and acceptance bar with a verdict-first FATAL/MAJOR/MINOR teardown. Keep the stage boundaries intact: novice playback is not teaching, whiteboard collaboration is not finished-artifact grading, and ownership audit is not exercise generation. Use `peer-review-writer` when the user reviews someone else's paper and `submission-readiness-audit` for a whole-paper author checklist.
- `skills/research-communication/` contains paper narrative, section writing, discussion-report synthesis, and talk-planning skills ÔÇö each scoped to one artifact (abstract, intro, method, figures, results prose, talk plan). The whole-draft writing coach that used to live here (paper-writer) is now **stage 10 of `flow-paper-lifecycle`**, which owns the four writing sub-phases (10a story/skeleton, 10b drafting, 10c math-prose pass, 10d handoff) and their gates, and carries the five `references/writing-*.md` files. Route "draft/outline the whole paper" there; route narrow section work to the singletons here.
- `skills/research-publishing/` contains reviewer-response, submission-readiness, venue-targeting, peer-review-writing, reproducibility-audit, and artifact-release skills. Note the author/reviewer split: `peer-review-writer` is for when the user is the REVIEWER of someone else's paper (claimÔÇôevidence map, weakness triage into fatal/fixable/minor, calibrated scores, meta-review and emergency-review modes, plus a PC-member / area-chair mode in `references/pc-member-guide.md` for bidding, COI, subreviewer management, and running discussion), while `reviewer-response-strategist` (author responding to reviews), `submission-readiness-audit` (author pre-submission), and `gap-finder` (mining a paper for one's own research) cover the author side; each disambiguates in its `description`.
- `skills/academic-career/` contains academic-career profile ingestion, PhD application writing, dossier scoring/eligibility evaluation, career support, and motivation skills. Note the deliberate split between `apply-package-auditor` (qualitative coherence/readiness audit, no scores) and `apply-dossier-evaluator` (reproducible 1ÔÇô5 rubric across 11 dimensions, eligibility gate, and scholarship-program fit for VEF/DAAD/Fulbright/Erasmus/MSCA); each disambiguates against the other in its `description`. `apply-dossier-evaluator` bakes in a strict volatility split ÔÇö stable structural logic only, with deadlines/cutoffs/score-minimums/word-limits/funding routed to a "verify against the live official call" instruction rather than committed as authoritative, and faculty/lab fit fetched live per run (no cached professor file).
- `skills/research-workflows/` contains lightweight router/meta-skills that sequence the singleton research skills into a guided workflow with stage gates, delegating the actual work to the singletons rather than duplicating it: `flow-learn-new-topic` (breadth-first onboarding into an unfamiliar area), `flow-deep-understanding` (depth-first mastery of one paper/method/theorem), `flow-idea-to-proof` (raw idea Ôćĺ formal claim Ôćĺ proof design), `flow-paper-lifecycle` (idea Ôćĺ experiments Ôćĺ writing Ôćĺ submission Ôćĺ rebuttal Ôćĺ artifact release), and `flow-phd-application` (profile Ôćĺ CV Ôćĺ fit Ôćĺ SOP Ôćĺ recommenders Ôćĺ outreach Ôćĺ package audit, plus a *conditional* route to `apply-dossier-evaluator` for graded scoring, cross-border eligibility, or scholarship-mission fit ÔÇö situational, not a linear stage). Orchestrators decide *what next* and which sub-skill to call; they never re-teach what a singleton already does.
- `skills/industrial-rnd/` contains skills for ambiguous industry problem framing, publication-grounded solution design, practical pilot/evaluation planning, and the banking/fintech domain specializations (the `banking-*` prefix groups them; there is no `banking/` sub-directory).
- `skills/presentation/` contains slide/deck *building* skills that render an approved per-slide spec into an artifact: `deck-beamer-academic` (Beamer for seminar/conference/lecture talks), `deck-beamer-proposal-report` (Beamer for proposals and progress/final reports), and `deck-business-report` (PowerPoint/Canva industrial/executive reports). Each builder is self-contained: the shared slide-spec schema and format-independent design rules were dissolved out of the former deck-design-principles skill into a per-builder `references/slide-spec-and-design.md` copy, so no builder depends on a sibling. These are downstream of `research-talk-planner`, which plans a talk's narrative and pacing but does not produce slides.
- `skills/coding-support/` holds `prompting-claude-models` only ÔÇö prompt/system-prompt/agent-scaffolding *design* for Claude-backed features (LLM-as-judge, RAG, extraction, multi-step agents), deliberately split into a durable technique layer in `SKILL.md` and a volatile model-specific layer deferred to the live Anthropic docs (catalogued in its `references/live-model-docs.md`) rather than reciting version-specific behavior that goes stale. It disambiguates against the built-in `claude-api` reference (API/SDK mechanics ÔÇö model IDs, pricing, parameters, tool-use schemas, caching) in its `description`; keep that boundary intact when editing either. The former Python-craft singletons (python-code-style, python-design-patterns, python-performance-optimization, python-testing-patterns) and debugging-strategies were removed: each was ~250ÔÇô350 words restating PEP 8, ruff defaults, and standard debugging practice that a capable agent already applies unprompted. **Do not re-add generic coding-craft skills.** The bar for this directory is material the model does not already default to.
- `skills/test-prep/` contains standardized-test preparation skills ÔÇö the `ielts-*` family and the `gre-*` family ÔÇö including `ielts-progress-reporter`, which aggregates a practice vault's marked `feedback/*.md` files into periodic progress reports (band trends, recurring errors, revision gains) rather than marking individual drafts. The IELTS scoring skills each carry their own `references/ielts-band-descriptors.md` scoped to that skill (`ielts-writing-task1`, `ielts-writing-task2`, `ielts-speaking-coach`, `ielts-learning-planner`, and `ielts-grammar-coach`); there is no shared band-descriptor file. `ielts-grammar-coach` is IELTS-first (its band-descriptor file is scoped to the **Grammatical Range & Accuracy** criterion only): it estimates a **GRA band**, then diagnoses and drills the specific recurring errors holding it down via coded corrective feedback (mark the error + type, force self-correction) rather than silent rewrite, Vietnamese-L1 contrastive diagnosis, a persistent `error-ledger.yaml` of systematic errors with spaced re-probes, and targeted drills (interactive by default, with a two-file offline set as an option); it also has a general-English mode (no band). Keep the boundary clear ÔÇö `ielts-writing-task1/2` and `ielts-speaking-coach` score the WHOLE task across all four criteria, whereas `ielts-grammar-coach` isolates GRA and drills recurring errors; and `ielts-progress-reporter` aggregates a vault into a report rather than marking or coaching. The `gre-*` family covers the shortened (post-Sept-2023) GRE with the same pick-one structure: `gre-verbal-reasoning` (Text Completion + Sentence Equivalence ÔÇö predict-then-match, trap taxonomy, miss classification), `gre-reading-comprehension` (passages + paragraph-argument critical reasoning ÔÇö passage mapping, wrong-answer taxonomy, line-reference discipline), `gre-quant-coach` (quantitative-comparison tactics, careless-error mechanism logs, content-gap triage; GRE test math only ÔÇö research-level math routes to `professor-mentor-technical-teaching`/`theory-*`), `gre-issue-essay` (the sole AW task, scored 0ÔÇô6 against its own `references/awa-scoring-rubric.md`, keyed to the six instruction stems), `gre-vocabulary-builder` (meaning clusters, charge, secondary meanings, spaced cloze re-probes; receives vocab-gap handoffs from `gre-verbal-reasoning`'s error log), and `gre-learning-planner` (allocation + daily vocab pipeline + POWERPREP milestones + test-date scheduling against application deadlines, with GRE-optional policies/percentiles/fees marked verify-live per the volatility rule). Keep the intra-family boundaries: blank-based items Ôćĺ `gre-verbal-reasoning`, passage-based Ôćĺ `gre-reading-comprehension`, word learning Ôćĺ `gre-vocabulary-builder`, scheduling Ôćĺ `gre-learning-planner`; and the cross-family boundary: GRE skills never handle IELTS tasks or vice versa (different tests, rubrics, and question logic).
- `skills/init-workspace/` is the top-level entrypoint for scaffolding non-coding, note-centric Obsidian workspaces: learning, ongoing research notes, research writing, academic applications, talks, test preparation, theory, and similar knowledge work. It must not claim generic software repositories, software applications, experiment codebases, websites, or other source-code projects. It asks one short intake batch (what the work is and what "done" means, project title plus an explicitly confirmed target directory, and what differs from the typical case); the user's answer confirms the target, so do not add a second confirmation round. It then chooses one of three paths: **use as-is** one of seven heavy templates under `templates/<type>/`; **adapt** the closest template, preserving its behavioral guarantees while updating Layout, routing, type labels, and all path-specific Operating-contract prose together; or **synthesize a lean bespoke vault** when none fits. Bespoke mode remains within the same non-coding, note-centric scope and always writes a folder tree plus `AGENTS.md`, while `references/` and `_dashboard/` are conditional on the intake. All paths write only into the target vault ÔÇö never into the repo's `templates/` ÔÇö and merge without overwriting on re-run. Templates: `ielts`, `paper`, `phd-application`, `research-notebook`, `talk`, `theory`, `learn-a-topic`. `research-notebook` is for ongoing research thought without a single paper or learning curriculum: `inbox/`, `questions/`, `ideas/`, `reading/`, `evidence/`, `decisions/`, `meetings/`, `syntheses/`, and `reviews/`; small captures skip the pipeline, while substantial syntheses use it. The `ielts` template ships AGENTS.md + dashboard only because installed `ielts-*` skills carry the band descriptors and feedback aggregation; the other six seed references. Each `AGENTS.md` opens with an agent-portable `## Operating contract`. Adapted contracts preserve the applicable invariants (non-coding knowledge work, source grounding or non-fabrication, marked uncertainty, faithful reporting and input checks where present, and the one-file pipeline rules where present) but must update type- and folder-specific wording so the contract matches the adapted Layout. A bespoke contract includes non-coding note work, non-fabrication, source grounding and uncertainty where claims are involved, report-faithfully behavior, and input preconditions. The six skill-routing templates (`paper`, `theory`, `learn-a-topic`, `research-notebook`, `talk`, `phd-application`) carry the input-precondition and report-faithfully rules; `ielts` (marking-only) carries neither. `talk` and `phd-application` additionally require grounding from the source rather than memory. The shared research pipeline remains generalized and agent-portable at `templates/_shared/agents/pipeline.md`: **evidence Ôćĺ draft Ôćĺ review Ôćĺ verify** inside one artifact file, with a size gate and no per-stage side files. Copy it into `<vault>/agents/` for `paper`, `theory`, `learn-a-topic`, `research-notebook`, and substantial bespoke source-grounded synthesis/research vaults; lighter or non-research vaults get no `agents/` folder. Every vault gets a one-line `CLAUDE.md` containing `@AGENTS.md`. Every `AGENTS.md` ends with synchronized Obsidian formatting rules: keep each prose paragraph or list item on one physical line and rely on Obsidian soft wrapping; require valid YAML frontmatter from line 1 and quoted wikilink lists in frontmatter; and forbid raw `|` in table cells. Keep this block synchronized across all seven templates.
- `templates/skill-template/` is for reusable starter material when creating new skills.
- `experiments/` is for shell scripts or prompt variants used to test skill behavior.
- `results/` is for saved evaluations or notes about which skill behavior worked.

## Skill taxonomy

The category directory names *are* the taxonomy ÔÇö there is no second list to keep
in sync. A new skill goes in exactly one of these:

| Directory | Scope |
|---|---|
| `research-discovery` | framing, positioning, gap-finding, novelty checks |
| `research-evidence` | literature mapping, Related Work, surveys, citation verification |
| `research-reasoning` | theorems, claims, proofs, derivations, formal audits |
| `research-experimentation` | experiment/ablation design, baselines, statistics, evaluation, run diagnostics |
| `research-engineering` | implementation recipes, codebase structure, reproducing a paper |
| `research-communication` | paper sections, figures, argument spine, talk planning |
| `research-publishing` | review, rebuttal, venue targeting, reproducibility, artifact release |
| `research-workflows` | the `flow-*` orchestrators that sequence the singletons |
| `learning-and-metacognition` | teaching, exercises, teach-back, co-solving, critique, knowledge-debt |
| `academic-career` | the `apply-*` family: profile, CV, SOP, fit, recommenders, outreach |
| `industrial-rnd` | industry problem framing, solution design, pilots, `banking-*` |
| `presentation` | the `deck-*` builders that render an approved slide-spec |
| `test-prep` | the `ielts-*` and `gre-*` families |
| `coding-support` | coding skills that clear the not-already-default bar |

Plus `skills/init-workspace/` at the top level, which is the project entrypoint
rather than a category member.

## Skill authoring conventions

- After adding, editing (renaming/moving/re-describing), or removing any skill, update all three registries so they stay in sync: this `CLAUDE.md` (structure/disambiguation notes), `README.md` (the skill catalog), and `skills.json` (the machine-readable index ÔÇö keep its `name`, `path`, `category`, and `description` correct, and verify it still parses as valid JSON).
- If the change introduces or revises a durable design principle, evidence-backed learning protocol, or cross-skill role boundary, update `DESIGN-PHILOSOPHY.md` as well. Keep operational skill instructions in `SKILL.md`; keep the philosophy file focused on rationale and references.
- Keep one skill focused on one clear capability.
- Keep each skill self-contained: a skill must not depend on resources outside its own directory, and skills must not share files or link into each other (e.g. no `../references/ÔÇŽ` cross-references). When several skills need the same reference material, give each its own copy scoped to that skill rather than pointing them at a shared directory.
- Put activation guidance in the frontmatter `description`; agents use this to decide when to load the skill. Do not use a separate `when_to_use` frontmatter key ÔÇö it is not read by agents. Merge all triggering guidance into `description`.
- Make descriptions specific and slightly pushy about when to use the skill, including common user phrasings.
- Group tight "pick-one" skill families with a short, domain-derived name prefix so siblings sort together and read as one suite: `apply-*` (PhD/career applications), `flow-*` (orchestrators/routers), `deck-*` (slide/deck builders), `theory-*` (theory-heavy math/ML tools), plus the existing `ielts-*`, `python-*`, and `banking-*`. Do not prefix the broad research singletons ÔÇö their directory already conveys the category, and prefixing all of them adds churn without aiding triggering (agents match on `description`, not name).
- Prefer procedural guidance: when to use the skill, when not to use it, the process to follow, and the expected output format.
- When two skills overlap, disambiguate in the frontmatter description and body. For example, field mapping belongs to `literature-triangulation`, polished Related Work prose inside an original paper belongs to `related-work-writer`, and standalone survey/tutorial/perspective papers belong to `survey-paper-writer`.
- Add `examples/` only when examples materially improve calibration.
- Add `scripts/` only when repeated manual work justifies a helper script.
- Add `evals/evals.json` for new skills when realistic test prompts will help evaluate triggering or output quality.
- Avoid broad scaffolding until there is real content for it.
- Remove low-priority or redundant skills instead of archiving them, unless they support the user's research, PhD application, or coding workflows.
- Do not vendor a `skill-creator` skill into this repo. Creating, editing, and evaluating skills uses Claude Code's built-in skill-creator, which ships the full eval/benchmark tooling; a repo-local clone only duplicates the `name`, shadows the built-in, and goes stale (a prior clone carried only `SKILL.md` while referencing scripts/agents/references it did not include).

## Commands

There is no build system, linter, test runner, or `package.json` in this
repository ÔÇö skills are consumed directly from git by the `skills` CLI, which
needs only `skills/` and `skills.json`. Installation wrappers live in
`scripts/install.sh` / `install.ps1`.

After any skill add / edit / move / removal, run these checks:

```bash
git status

# 1. Frontmatter must PARSE AS YAML, not merely look right.
#    A plain scalar containing ": " is invalid and the skills CLI drops the
#    skill SILENTLY ÔÇö no error, it just vanishes from the listing. Use `>-`
#    block style for any description containing quotes, colons, or examples.
python3 -c '
import os,re,yaml
for r,_,fs in os.walk("skills"):
    if "SKILL.md" not in fs: continue
    m=re.match(r"^---\n(.*?)\n---",open(os.path.join(r,"SKILL.md")).read(),re.S)
    fm=yaml.safe_load(m.group(1))
    assert fm["name"]==os.path.basename(r), r'

# 2. Registries must agree: skills.json <-> disk <-> README catalog, all 3 equal.
# 3. Depth: no SKILL.md more than two levels under skills/ (see Structure above).
```

The authoritative end-to-end check is the CLI itself, which catches what static
checks miss. Against a local clone of the working tree:

```bash
npx skills add "file:///path/to/clone" -l    # must list EVERY skill, no flags
```

If executable helper scripts are added under `experiments/` or skill-local
`scripts/`, document the exact run commands here when they become stable.
